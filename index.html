VLA-Flight

Instruction-Grounded Drone Navigation with Vision-Language-Action and Onboard IMU

A lightweight ~80M parameter VLA architecture that jointly encodes RGB images, onboard IMU data, drone position, and natural language instructions to predict low-level flight control commands — running entirely onboard a Jetson Orin NX 16GB at >30Hz. Target application: autonomous agricultural drone navigation.

Project Background

What is VLA-Flight?

Existing drone VLA models (OpenVLA, RaceVLA, AerialVLA) rely on massive 7B+ parameter LLMs, require server-grade GPUs, and cannot run onboard. VLA-Flight replaces the LLM with lightweight cross-attention fusion to enable real-time onboard inference for agricultural drone navigation.

Key Contributions

  • C1: Dual IMU Usage — First drone VLA to fuse raw IMU at two levels: direct temporal fusion (fine dynamics like wind/drift) and fuzzy directional prompting (coarse intent like "forward-right" prepended to instruction text). AerialVLA only uses the second.
  • C2: LSAC + VAE Action Chunking — Predicts H=10 step action chunks for smooth trajectories, executes only h=5 before replanning. VAE bottleneck prevents mode-averaging when multiple valid paths exist (e.g., go left or right around obstacle).
  • C3: Edge-Deployable (~80M params) — Replaces 7B LLM with frozen MiniLM (22M) for instruction encoding + cross-attention fusion decoder (~5M). Continuous action regression instead of 256-bin discretization. 100x smaller, 10x faster than OpenVLA.
  • C4: Geometric Safety Correction — Zero-parameter post-processing: velocity clamping, altitude bounds, acceleration smoothing, depth-based repulsion, IMU vibration proximity detection. Adds zero inference latency.
  • C5: Progressive 3-Stage Training — Stage 1: perception pretraining (vision + IMU). Stage 2: navigation SFT (full end-to-end). Stage 3: agricultural domain refinement.

Architecture Overview

Data Pipeline

Problem 1: MUN-FRL Batch Download Broken

The download script (scripts/download_datasets.sh) was failing silently for MUN-FRL. Every downloaded .bag file was empty — the script was saving the HTTP response header instead of the actual file content.

What Happened

  • The batch download script pulled files from the MUN-FRL server
  • Files appeared to download successfully (no errors)
  • But every .bag file was essentially empty — just a few bytes containing the HTTP header, not the actual ROS bag data
  • Preprocessing would then fail or produce garbage because there was no real sensor data in the files

The Fix

  • Manually downloaded each ROS bag file directly from mun-frl-vil-dataset.readthedocs.io
  • Verified file sizes matched expected values (~GB per bag, not KB)
  • Confirmed bags contained valid ROS topics with rosbag info before preprocessing
  • Tedious but necessary — automated script needs a proper fix for redirect handling

Problem 2: Hardcoded ROS Topics in Preprocessing

The preprocessing script (scripts/preprocess_dataset.py) was written for UZH-FPV and had all ROS topic names hardcoded. MUN-FRL uses completely different topic names, message types, and data formats. The script would silently find zero messages and produce empty output.

Additional fix: IMU-to-Image synchronization. MUN-FRL records images at 20Hz and IMU at 400Hz. Had to implement ApproximateTimeSynchronizer to pair the two streams and interpolate IMU readings to align with image timestamps. For actions, computed velocity labels as position deltas from PPK ground truth: v = (pos[t+1] - pos[t]) / dt.
Training

Problem 3: NaN Loss in Stage 2 Training

Training consistently crashes with NaN loss approximately 10 epochs into Stage 2 of the progressive training pipeline. Stage 1 (perception pretraining) completes fine. The crash happens the moment the Fusion Decoder is unfrozen.

Root Cause Analysis

  • Stage 1 trains vision + IMU encoders separately — simple, stable math. Works fine.
  • Stage 2 unfreezes the Fusion Decoder — this is the cross-attention layer where vision and IMU embeddings are multiplied together. Very sensitive to weight magnitudes.
  • The initial learning rate of 1×10-3 caused the cross-attention scores to explode. Large attention weights → large softmax inputs → extreme gradients → weights spiral out of control.
  • fp16 precision has a max value of ~65,504. Much smaller than fp32's ~3.4×1038. The exploding values overflow to Inf, and Inf - Inf = NaN, which poisons every subsequent computation.
  • Required for deployment — Jetson Orin NX needs fp16 to hit >30Hz, so switching to fp32 is not an option.

Initial Fix

  • Reduced learning rates to give the fusion decoder time to slowly align the vision and IMU signal spaces without oscillating:
# config: progressive training LRs

# Stage 2: Navigation SFT (fusion decoder unfrozen)
stage2_lr: 1.0e-3 → 2.0e-4

# Stage 3: Domain refinement (agricultural)
stage3_lr: 1.0e-3 → 3.0e-4
Result: Training no longer crashes at epoch 10. Loss decreases smoothly through Stage 2. But the optimal LR is still unknown — 2×10-4 might be too conservative (slow convergence) or still too aggressive for some dataset combinations.
Action Plan

How I'm Addressing the Open Issues

NaN / Learning Rate Tuning

  1. Learning rate sweep: Run a grid of LRs around the current fix — both higher (5×10-4) and lower (1×10-4) — to find the stability boundary for fp16 training with the fusion decoder.
  2. Monitor gradient norms: Add gradient norm logging per-layer to detect when cross-attention starts diverging, before it hits NaN. This will tell us the exact tipping point.
  3. Consider gradient clipping: If the sweep shows a narrow stable window, add max-norm gradient clipping as a safety net so we can use a higher (faster) LR without overflow risk.

Instruction Diversity

  1. Problem: Every episode currently uses the same generic instruction.txt (e.g., "Fly through the racing track"). The model can't learn to ground actions on language if every instruction is identical — it just ignores the text encoder.
  2. Fix: Generate per-trajectory instructions that describe the actual content of each episode — direction changes, obstacle types, terrain features. Will vary vocabulary and sentence structure.
  3. Why it matters: The instruction encoder (MiniLM) is frozen. If every instruction embeds to the same vector, the cross-attention fusion learns to ignore it entirely. Diverse instructions force the model to actually attend to language.
